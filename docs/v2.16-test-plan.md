# v2.16 — manual test plan

Reference checklist for the operator validating an upgrade. The
automated `php artisan test --testsuite Unit` and `Feature` suites
should already be green; this document covers the user-visible
acceptance scenarios that are hard to assert from a unit test.

Each scenario references the PR that introduced it so a failure can
be traced back to a single commit window.

## Setup

```bash
git fetch && git checkout v2.16
php artisan clientxcms:upgrade
```

Then sign in as a customer and as an admin on two browser tabs.

## Smoke tests

### S1. Renewal does not duplicate invoices (PR #1)

1. Sign in as a customer holding an active monthly service.
2. From `/client/services/{uuid}` click *Renew*.
3. Refresh the page and click *Renew* again two times quickly.
4. Open `/client/invoices` — only one pending invoice should be
   listed.

### S2. Admin form preserves typed input (PR #2)

1. As admin, open `/admin/customers/create`.
2. Fill the form, intentionally type an already-used email.
3. Submit — the page returns with a validation error.
4. Every other field must keep the value you typed (not the model's).

### S3. Branded error pages (PR #3)

1. Hit `/some-route-that-doesnt-exist` — see the redesigned 404.
2. Sign out then hit `/admin/dashboard` — see the redesigned 401 or
   the redirect to the login page (depending on the auth middleware).
3. Tail `storage/logs/laravel-*.log` after each request — no "view
   not found" or similar error.

### S4. Phone widget with international flag (PR #4)

1. As anonymous, visit `/register`.
2. The phone field shows a 🇫🇷 flag + `+33` prefix on its left.
3. Change the country `<select>` — the flag follows.
4. Submit the form with a national number (e.g. `06 12 34 56 78`).
5. Re-open `/admin/customers/{id}` — the customer row holds the
   number in E.164 format (`+33612345678`).

### S5. Avatar upload (PR #5)

1. `/client/profile` → *Upload photo* with a 500×800 jpg.
2. Confirm the resulting avatar in the header dropdown is square
   (centre-cropped to 256×256).
3. Remove the avatar → coloured initials reappear.

### S6. Accessibility skip-link (PR #6)

1. Load any page, hit `Tab` once with a fresh focus — a black
   "Skip to main content" link appears in the top-left.
2. Press Enter — caret jumps to `#content`.

### S7. Helpdesk editor (PR #7)

1. As admin, open a ticket.
2. Take a screenshot with Windows+Shift+S (or Cmd+Shift+4 on Mac)
   and Ctrl+V into the reply editor — the file is attached with
   name `pasted-{date}.png` and a green flash appears.
3. Drag a file from the desktop onto the editor — same chip
   behaviour.
4. Click *Preview* → fetch hits `POST /helpdesk/preview`, returns
   the server-rendered Markdown (headings + lists rendered).
5. Submit the reply — buttons disable for 6 s, no duplicate reply.

### S8. Ticket access rules (PR #8)

1. As admin, `mysql> INSERT INTO support_access_rules (name,
   predicates, message_key) VALUES ('No high priority',
   '{"block_priorities":["high"]}', 'v216::helpdesk.access.high_priority_restricted');`
2. As customer, try to open a high-priority ticket — error message
   appears, ticket not created.
3. Drop the rule, try again — passes.

### S9. SLA tracking (PR #9)

1. As admin, set a department's `sla_first_response_minutes = 60`.
2. As customer open a ticket against that department.
3. `mysql> SELECT first_response_due_at FROM support_tickets WHERE
   id = LAST_INSERT_ID();` — within 60 minutes of `created_at`.
4. As admin reply → `first_response_at` is set to now.
5. `php artisan helpdesk:notify-sla-breach --dry-run` lists nothing
   (in time).

### S10. Permissions scoped (PR #10)

1. Create a role with `admin.manage_tickets` scoped to
   `(department, 3)`.
2. Assign it to an admin.
3. That admin can view tickets of department 3 but not other
   departments.

### S11. SMS MFA driver (PR #11)

1. Set `mfa_sms_driver=log`.
2. From the SmsService gateway → tests pass.
3. Send an SMS — confirm `mfa.sms.log_driver` is the only line in
   the log; the OTP body itself never appears.

### S12. GDPR export + purge (PR #12)

1. As customer click *Export my data* on `/client/profile`.
2. The flash message shows a signed link valid 24 h.
3. Download — ZIP contains manifest.json, profile.json, invoices,
   services, tickets, api_tokens (names only).
4. Switch to a second customer account, try the same signed URL —
   403.

### S13. Billing — atomic numbering + SHA-256 (PR #13)

1. Issue 3 invoices in parallel from 3 browser tabs (e.g. via
   "Renew" on 3 different services).
2. `mysql> SELECT invoice_number, pdf_sha256 FROM invoices ORDER
   BY id DESC LIMIT 3;` — all 3 different numbers, all 3 hashes
   filled.

### S14. Pay-as-you-go billing (PR #14)

1. As admin add a `ProductMeteredRate` row for a metered product:
   `metric_key=cpu_cores, unit_price=0.50, included_quantity=2`.
2. As provisioning node POST to `/api/services/{uuid}/metrics`
   with `{"metrics":[{"key":"cpu_cores","value":5}]}`.
3. Run `php artisan usage:bill-cycle --period=current-month`.
4. The invoice line should read `5 - 2 = 3 cores × 0.50 = 1.50€`.

### S15. Service live status (PR #15)

1. As customer open `/client/services/{uuid}`.
2. Open DevTools → Network. The page polls
   `/client/services/{uuid}/status` every 15 s.
3. Switch to another browser tab — polls stop within ~30 s.
4. Switch back — polls resume immediately.

### S16. Promo code on /basket/add (PR #16-basket)

1. As customer visit `/basket/config/{product}`.
2. Expand "I have a promo code" → enter a code.
3. The page reloads on the same URL, summary shows a green
   "Coupon — CODE  -X.XX €" line, total updates accordingly.
4. Click "Remove" → coupon dropped, total reverts.

### S17. Extension sandbox (PR #17)

1. Create a broken provider:
   ```php
   namespace App\Demo;
   class BrokenProvider extends \Illuminate\Support\ServiceProvider {
       public function register(): void { throw new \RuntimeException('boom'); }
   }
   ```
2. Register it via an extension. Hit any page — page renders, the
   extension is flipped to `enabled=false` in
   `bootstrap/cache/extensions.json` and a `boot_error` entry is
   recorded. `storage/logs/` carries the `extensions.boot_failed`
   line.
3. `APP_SAFE_BOOT=true` env var → every extension is skipped on
   the next request.

## Pass criteria

- All 17 scenarios above pass without manual SQL fixes.
- `php artisan clientxcms:check` exits zero.
- `php artisan test` Unit + Feature suites exit zero.
- `/health`-style probes return HTTP 200 (`curl -sI https://your-host/`).
